Exchange Mechanics - Week 1

Overview of Exchange Operations

This week, we'll dive into the core operations that make up our exchange, focusing on the mechanics behind order matching and how trades are executed within our limit order book simulator.

Types of Orders

  • Limit Orders: Orders to buy or sell at a specified price or better.
  • Market Orders: Orders to buy or sell immediately at the best available price.
  • Stop Orders: Orders that become executable only when a certain price level is reached.
We'll be focusing on limit orders for our exchange.

Order Matching and Execution

Our exchange will use a price-time priority mechanism for order matching. This means that orders are first ranked by their price; for orders with the same price, the order that arrived first has priority.

There are many approaches to this... I'm going to leave figuring out how exactly to do it as an exercise for the reader.

One way or another, you'll likely want to overload the rich comparison operators in the Order object. (Google "python compare objects by attributes")

Repo Structure

The repo containing this spec also will contain the project! Git clone this bad boy to get started.
  • We're gonna be using the MVC (Model, View Controller) structure for this project. In software engineering, you'll run into the problem of coupling - functions/methods that overly depended on each other to the point where any change to repo becomes a breaking change. The MVC structure is a great way to mitigate this problem for certain software applications.
  • The model directory will contain all of our data models (Exchange & Order).
  • The controller directory will contain all of our functions that alter our data models.
  • The view directory is ... currently not there. But if it was, it would contain all the functions that render our application's information.
  • You can feel free to ignore the docs folder, that just contains the html files related to the project spec.
  • The resources directory contains JSON that you can use to test your code. Each json file should contain a "orders" array which contains more JSON objects containing information relevant to an order.
  • The test directory should contain all of your testing. You'll notice that it's empty - you'll be writing your own tests! It's important to know how to test your software as an engineer and how to write your software to be testable. Also it'll come up in interviews :P.
